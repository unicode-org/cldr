On 2003.12.19, the Version 1.0 Final Candidate was made available. The CLDR Version 1.0 final release is pending approval by the OpenI18N steering committee.

The purpose of the Common Locale Data Repository (CLDR) project is to:

  1. devise a general XML format for the exchange of culturally sensitive (locale) information for use in application and system development

  2. gather, store, and make available data generated in that format

CLDR 1.0 is just the first step. It provides a version of the data that can be referenced, but enhancements and fixes will come in future versions. We request that people continue to review the data for languages and countries that they are familiar with for the next version of CLDR, and file bug reports where they see problems.

The easiest way to provide feedback is to view the comparison charts, which provide an HTML view of data from different sources. There is also locale-insensitive supplementary data which is only available in XML format. That data tells which country uses which currency, how many digits each currency takes, and any special rounding behavior.

XML Source. The locale data XML specification and locale data XML files are checked into a CVS repository, currently hosted on the ICU site, and available via the web or by using CVS clients. The CVS tag is "release-1-0-candidate". For information on using CVS clients to access the repository, see readme.html.

The XML source data is organized in the following directories. Within those directories, ignore the folders icu_format/; those are just for production. The xml files can be divided into main/ and collation/.

aix/
common/
hp/
ibmjdk/
icu/
linux/
open_office/
solaris/
sunjdk/
windows/

Note: the data for all but Common is provided for comparison only, and is not to be viewed as authoritative or referenceable.

Known Issues

  • The XML and HTML files generated as part of this work appear to stress the common browsers. Most of them will fail on at least some of the XML files, even though we have verified the formats with the Xerxes parser. If that happens, try opening with the menu View > Source, or its equivalent.

  • The generated xml files are not minimal. That is, elements might be repeated that would be inherited from the parent.

  • The comparison collation data is only generated for a few data sources at present.
